# Repo Orientation: Nova

> Generated by orient. Re-run to update.

## One-line purpose
Nova is a terminal-based (TUI) AI coding agent written in Zig ‚Äî "the coding agent for shipping to the stars" ‚Äî built around the assumption that a human is always in the loop, driving an LLM that can read, run commands, and edit a codebase.

## Primary language(s)
- **Zig** (0.16) ‚Äî all application code in `src/` and `lib/`.
- **C** (vendored) ‚Äî SQLite (`vendor/sqlite`) and the `fff` file-finder (`vendor/fff`), pulled in via `@translate-c` (`src/c.h`).
- **Rust** (third-party only) ‚Äî the upstream `fff` source lives in `third_party/fff`; Nova consumes its compiled C ABI, not the Rust.

## Pipeline / workflow stages
A user prompt flows through the system like this:

1. **TUI input** (`src/tui.zig`) ‚Äî the user types in the libvaxis vxfw interface; commands (`/...`) and `@`-mentions are recognised here.
2. **Hand-off to the worker** (`src/tui/agent_worker.zig`) ‚Äî the prompt is handed to a background thread so the UI never blocks on I/O or network.
3. **Agent turn loop** (`src/agent.zig` ‚Üí `Agent.run`) ‚Äî appends the user message, then repeatedly calls the model until there are no more tool calls (capped at 100 iterations).
4. **LanguageModel call** (`src/ai.zig` ‚Üí `LanguageModel.prompt`) ‚Äî dispatches to one provider adapter (codex-responses, openai-compatible, or openai-responses) and streams the reply back via observer callbacks.
5. **Tool execution** (`src/executor.zig` + `src/tools.zig`) ‚Äî any tool calls the model emits (currently just `bash`) are run, and results are folded back into the conversation.
6. **Event stream back to the UI** ‚Äî the agent emits `Agent.Event`s into a bounded queue (`src/tui/agent_worker.zig`); the UI thread drains them each frame and re-renders.
7. **Persistence** (`src/session.zig` + `src/db.zig`) ‚Äî every message and tool call is written to a SQLite-backed session so conversations are resumable and branchable.

## Key files
- `src/root.zig` ‚Äî library entry point (`run`); wires up config, logger, the `AgentRuntime`, and launches the TUI. Read this first to see how everything is assembled.
- `src/main.zig` ‚Äî thin executable shell: sets the allocator and panic handler, then calls `nova.run`.
- `src/agent.zig` ‚Äî the heart of the agent. `Agent.run` is the turn loop; also owns the steering message queue and the `Agent.Event`/`Listener` emitter pattern.
- `src/ai.zig` ‚Äî the inference seam. Defines `LanguageModel` (a union over provider adapters), `ChatMessage`, content blocks, and the streaming observer types.
- `src/ai/openai_compatible.zig`, `src/ai/openai_responses.zig`, `src/ai/codex_responses.zig` ‚Äî the three provider adapters behind the `LanguageModel` union.
- `src/runtime.zig` ‚Äî `AgentRuntime` owns the live client, system prompt, skills, and session writer; manages connect/disconnect and client lifetime.
- `src/tui.zig` ‚Äî the entire vxfw UI (~2000+ lines): conversation rendering, command panel, model/provider pickers, diff counts, status bar.
- `src/tui/agent_worker.zig` ‚Äî the threading boundary: the bounded `EventQueue` and `runAgentTurn`, which runs the agent off the UI thread.
- `src/tools.zig` + `src/tools/bash.zig` ‚Äî the tool registry (single source of truth for what tools exist) and the `bash` tool with its agent-friendly middleware.
- `src/session.zig` + `src/db.zig` ‚Äî SQLite-backed persistence and the conversation tree (resumable, branchable sessions).
- `build.zig` ‚Äî the build graph: every module, the vendored C sources, and the `run`/`test` steps. Essential for understanding how the pieces are wired.

## Core concepts
**Agent as single emitter (`src/agent.zig`)**: The `Agent` is the *only* thing that emits public events (`Agent.Event` / `Agent.Listener`). Sub-modules (the streaming layer, the executor) report progress up through narrow observers ‚Äî `StreamObserver`, `ToolCallObserver` ‚Äî never directly to the UI. This keeps a clean one-way data flow and keeps payloads C-flattenable for a possible future FFI shim.

**LanguageModel union (`src/ai.zig`)**: The inference boundary. `LanguageModel` is a tagged union over the provider adapters; calling `.prompt(...)` dispatches to whichever provider is connected. New providers are added as new union members. Nova accepts any OpenAI-compatible endpoint (`/completions` or `/responses`) plus Codex.

**The two-thread model (`src/tui/agent_worker.zig`)**: The UI thread and the agent worker thread are separate. They communicate through a bounded `EventQueue`. The worker *blocks and retries* on a full queue rather than dropping events ‚Äî a lost `turn_finished` would strand the UI's `in_flight` flag forever and look like a hang.

**Steering via a bounded queue (`src/agent.zig`)**: While the agent is mid-turn, new user messages are pushed onto a bounded message queue rather than interrupting. After each tool-calling turn finishes, the queue is drained and the messages appended to the conversation.

**The conversation tree (`src/session.zig`, `docs/ARCHITECTURE.md`)**: A session is a *tree*, not a linear log. Users can branch off at any user message, assistant message, or even tool call to pursue a different path; every branch is persisted to SQLite and resumable.

## Common gotchas
- **Zig version & idioms matter.** This is Zig 0.16. `ArrayList`/`HashMap` are used unmanaged (allocator passed per-call: `list.append(allocator, x)`), and the new `std.Io` writer interface is everywhere. See `AGENTS.md` for the exact current patterns before writing any Zig.
- **Don't put display policy in the tool/agent layer.** Expand-by-default, render mode, and colour rules live in the TUI (`src/tui.zig`, `src/tui/style.zig`), not in `src/tools.zig` or `src/agent.zig`. The `Render` enum (`{ plain, diff }`) in `src/thread.zig` describes *how* to draw, but the TUI owns the *policy*.
- **Never block the UI thread.** Any file I/O or network work must happen on the agent worker thread (see `runAgentTurn` and `addUserPrompt`, which expands `@`-mentions on the worker thread on purpose). Adding I/O to a UI-thread code path will cause visible stalls.
- **Tests are registered centrally.** Inline tests must be reachable from `refAllDecls` in `src/root.zig` / `src/main.zig` or they won't run under `zig build test`.

## Suggested exercise sequence

**Exercise 1 ‚Äî Trace the entry point.**
Open `src/root.zig` and read the `run` function top to bottom (it is short). Then close it and explain, in your own words: what are the major things Nova sets up before the TUI starts drawing, and in what order? Specifically, where does the system prompt come from, and what object ends up owning the live model client?

**Exercise 2 ‚Äî Understand the turn loop.**
Open `src/agent.zig` and read just the `run` function (around line 205, ~40 lines). After reading, answer: when the model returns *no* tool calls, what does the loop do before deciding the turn is over ‚Äî and why does that step exist? (Hint: connect your answer to the "Steering" section of `docs/ARCHITECTURE.md`.)
